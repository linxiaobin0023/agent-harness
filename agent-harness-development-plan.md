# Agent Harness 全流程开发方案

## 1. 项目目标

本项目要实现一套“全流程 Agent Harness 系统”，核心目标不是单纯让 AI 写代码，而是通过工程化约束把 AI 变成可控、可审查、可迭代的开发执行体。系统需要覆盖从“想法输入”到“需求文档、设计规范、设计图、开发计划、代码实现、自动审查、自动修复、提交推送、Phase 验证、迭代修改、构建发布、经验沉淀”的完整链路。

本方案强调三件事：

1. **前馈引导**：在开发前把任务讲清楚、约束讲清楚、产出物讲清楚。
2. **反馈闭环**：在开发后自动检查、自动修复、自动复核。
3. **持续进化**：把每次执行中的失败模式、低分规则和重复操作沉淀为新规则或新 Skill。

---

## 2. 核心设计原则

### 2.1 Harness Engineering

系统的本质是“搭建约束与反馈环境”，而不是完全依赖模型自觉。

- **约束**：每个阶段都有明确输入、输出、边界和验收条件。
- **引导**：通过 Skill 让 AI 在动手前先完成需求澄清、方案拆解和设计对齐。
- **反馈**：通过 Hook、Review、自动验证脚本对结果做检查。
- **修正**：发现问题后自动进入修复流程，而不是一次性输出后结束。

### 2.2 阶段化产物优先

所有关键工作都必须产出结构化文档或明确结果，而不是直接跳到代码。

推荐顺序：

1. 想法描述
2. 需求文档
3. 设计规范
4. 设计图 / 交互稿
5. 开发计划
6. 项目开发
7. 自动审查
8. 自动修复
9. 提交与推送
10. Phase 验证
11. 迭代修改
12. 构建发布
13. 经验沉淀

### 2.3 隔离优先

执行层采用多个独立 Sub-Agent 实例，每个实例都只处理自己的职责，且**不继承彼此上下文**。

这样做的目的：

- 防止错误上下文传播
- 减少任务污染
- 便于定位问题来源
- 让 Review 更接近真实独立审查

## 3. 总体结构

系统按“调度层 → 执行层 → 引导层 → 检查层 → 进化层”组织，和仓库里的项目结构保持一致。

### 3.1 调度层：`CLAUDE.md`

`CLAUDE.md` 是毒舌产品经理式的总调度入口，负责控制“谁做什么、什么时候做、做到什么程度”。它不直接实现业务，而是负责：

- 项目状态检测
- 流程路由
- Skill 调度
- Sub-Agent 派发
- Review 与 Stop Gate 控制

这一层是整个 Harness 的大脑，所有后续动作都要经过它的调度。

### 3.2 执行层：Sub-Agent × 4（Context Firewall）

执行层包含 4 个独立 Sub-Agent，每个 Task 都使用全新的实例，不复用、不继承历史上下文。

1. **implementer**：编码实现、编译验证、自检
2. **code-reviewer**：两阶段审查，先 Spec 合规，再代码质量
3. **feedback-observer**：捕捉用户反馈，写入 feedback 文件
4. **evolution-runner**：扫描 feedback 累积，生成进化建议

这层的目标是保证每个 Task 都是干净执行，防止上下文污染。

### 3.3 引导层：Skills × 8（Guides / 前馈控制）

Skill 负责把“人类意图”转成“可执行任务结构”，并在行动前注入标准和方法论，提高一次做对的概率。

1. **product-spec-builder**：需求收集 + 迭代更新
2. **design-brief-builder**：视觉规范 + 配色 / 交互量化
3. **design-maker**：设计图制作（Pencil / Figma MCP）
4. **dev-planner**：Phase 拆分 + 技术调研
5. **dev-builder**：Task 级编码 + 编译验证
6. **bug-fixer**：四阶段系统性调试
7. **code-review**：两阶段审查（Spec + Quality）
8. **release-builder**：构建打包 / 部署发布

### 3.4 检查层：Hooks × 6 + Review 闭环（Sensors / 反馈控制）

Hook 是确定性传感器，负责在关键节点自动检查，避免模型“忘了做”。

- `pre-commit-check`：编译不过阻止提交
- `auto-push`：commit 后自动推送
- `stop-gate`：代码未审查不允许停
- `detect-feedback-signal`：检测用户修正信号
- `mark-review-needed`：代码变更标记为待审查
- `check-evolution`：session 启动时检查反馈积累

Review 闭环为：**Review → Fix → Re-Review → Pass**。

### 3.5 进化层：`.claude/feedback/` + `EVOLUTION.md`（Steering Loop）

进化层负责把执行经验变成可复用的规则与新能力。

- `feedback` 文件：记录用户反馈与审查结果
- `FEEDBACK-INDEX.md`：索引与快速加载
- `EVOLUTION.md`：四层进化路径与升级规则

目标包括：

- 同类错误不重复犯第二次
- 高频反馈升级成正式规则
- 低分 Skill 持续优化
- 新的重复模式沉淀成新 Skill

## 4. 核心功能模块

### 4.1 `product-spec-builder`

输入：用户想法、目标、约束、参考案例。

工作方式：通过多轮追问把模糊需求拆解为结构化约束，尽可能问到使用场景、范围边界、交互方式、项目管理方式、内容结构、异常情况、验收标准等细节。

输出：

- 目标说明
- 范围边界
- 非目标
- 依赖项
- 验收标准
- 风险点
- 变更记录

说明：这份 Product Spec 是给后续所有 Skill 当执行基准的，不是给人随手阅读的说明书。

### 4.2 `design-brief-builder`

输入：Product Spec、视觉偏好、品牌约束、参考设计。

工作方式：把抽象风格偏好量化成可以稳定复用的视觉规则，避免“深色极简”被不同模型理解成不同东西。

输出内容建议包括：

- 配色方案
- 点缀色性格
- 字体与排版原则
- 文档结构
- 交互元素风格
- 动效级别
- 响应式要求
- 可访问性要求

### 4.3 `design-maker`

输入：Product Spec、Design Brief、页面结构与流程。

工作方式：通过 Pencil 或 Figma 的 MCP 连接，直接在设计工具中生成完整页面原型图。

规则：

- 设计图的权重最高
- 后续实现优先参照设计图，其次 Design Brief，最后才是 Product Spec
- 没有设计图时，不应进入完整 UI 实现阶段

### 4.4 `dev-planner`

输入：Product Spec、Design Brief、设计图、当前项目结构。

工作方式：先调研技术栈和现成组件，再拆分 Phase。

输出：

- Phase 拆分
- 每个 Phase 的交付内容
- Task 列表
- 依赖关系
- 风险与缓解措施
- 验收点
- DEV-PLAN 文档

说明：DEV-PLAN 同时也是跨 session 续接的进度锚点，新开对话时可直接读取它继续工作。

### 4.5 `dev-builder`

输入：DEV-PLAN、当前任务、涉及文件、相关约束。

工作方式：不一口气写完整个 Phase，而是按 Task 逐个完成。

规则：

- 每完成一个 Task，先编译验证
- 通过后再进入下一个 Task
- 出问题时可以精确定位到引入问题的 Task
- 不允许攒到最后再统一修

### 4.6 `bug-fixer`

输入：报错信息、stack trace、最近变更、复现步骤。

工作方式：使用四阶段系统性调试法，不猜不试，不靠直觉乱改。

#### 四阶段系统性调试

1. **收集证据**：读完整错误信息和 stack trace，复现 bug，检查最近变更，追踪数据流。
2. **分析模式**：找相似正常功能做对比，识别差异和可疑点。
3. **假设验证**：形成最多 3 个假设，按可能性排序，用最小改动验证。
4. **实施修复**：一次只改一个逻辑点，改完做编译验证、功能验证、回归验证。

核心原则：**不猜不试，一次只改一个地方**。

补充规则：

- 如果同一个 bug 反复修多次仍未解决，就停止硬磕，重新审视问题本身
- 可能是架构问题、环境问题、理解偏差，而不只是代码问题

### 4.7 `code-review`

输入：实现代码、Product Spec、Design Brief、设计图、DEV-PLAN。

工作方式：两阶段审查，且是最重要的推理型传感器。

#### Stage 1 功能完整性审查

逐条对照 Spec 和计划，检查：

- 有没有漏实现
- 有没有多做
- 行为是否偏离需求
- 关键流程是否闭环

规则：Stage 1 出现 HIGH priority 问题时，直接停在 Stage 1，不进入 Stage 2。

#### Stage 2 代码质量审查

检查：

- 命名
- 类型
- 结构
- 安全
- 可维护性
- 测试完整性

自动闭环：

功能开发完成 → `code-reviewer` 执行 Stage 1 → Stage 1 通过进入 Stage 2 → Stage 2 通过自动 commit + push → Stage 2 失败调用 `bug-fixer` 修复 → 再审查 → 循环直到通过。

### 4.8 `release-builder`

输入：已通过审查的代码、发布目标、环境配置。

输出：

- 构建产物
- 发布说明
- 环境配置说明
- 验证步骤
- 回滚预案

## 5. Hooks × 6 + Review 闭环

系统的另一半是反馈控制。Hook 作为确定性传感器，在关键节点自动触发检查，不依赖模型自己“记得”去做。

### 5.1 `pre-commit-check`

每次 commit 之前自动跑编译检查。编译不过，commit 直接被阻止。

### 5.2 `auto-push`

commit 成功后自动推送到远程仓库，不需要手动操作。

### 5.3 `stop-gate`

Agent 准备停下来的时候检查当前状态：如果代码已修改但还没做 code review，不允许结束。

### 5.4 `detect-feedback-signal`

自动检测用户消息里的修正或不满关键词，例如“你搞错了”“不是这样”“你又忘了”。检测到后提醒主 Agent 记录反馈。

### 5.5 `mark-review-needed`

代码文件被编辑或创建后，自动标记为“需要 review”，配合 `stop-gate` 使用。

### 5.6 `check-evolution`

每次新开 session 时检查 feedback 积累，有待处理内容就提醒。

### 5.7 Review 闭环

功能开发完成后必须进入 `code-reviewer`：

- 功能实现完成 → `code-reviewer` 执行 Stage 1
- Stage 1 通过后进入 Stage 2
- Stage 2 通过后自动 commit + push
- Stage 2 失败则调用 `bug-fixer` 修复后重新审查
- 循环直到两个 Stage 都通过

这套闭环是最重要的推理型传感器，用来持续校准“实现是否真的符合需求”。

## 6. `.claude/feedback/` + `EVOLUTION.md`（Steering Loop）

进化层负责把执行经验变成可复用的规则与新能力。

### 6.1 `feedback` 文件

`feedback-observer` 在后台记录反馈，尽量不打扰用户。反馈可以是：

- 用户明确纠正
- Review 发现的问题
- 重复出现的失败模式
- 低评分 Skill 的证据

### 6.2 `FEEDBACK-INDEX.md`

提供反馈索引与快速加载能力，方便新 session 迅速知道历史问题是什么。

### 6.3 `EVOLUTION.md`

定义四层进化路径与升级规则：

1. 记录
2. 毕业
3. 优化
4. 创建新 Skill

### 6.4 四层进化路径

- **第一层：记录** —— 静默收集反馈，后台完成
- **第二层：毕业** —— 同类反馈出现三次以上，提议升级为正式规则
- **第三层：优化** —— 某个 Skill 持续低分，提议优化该 Skill
- **第四层：创建新 Skill** —— 某种重复模式没有现成 Skill 覆盖时，提议创建新的 Skill

规则：

- 记录与扫描在后台完成
- 只有产生具体建议才提示用户
- 每条建议都需要用户确认后才执行
- 绝不自动改规则

目标是让系统从使用中持续学习，并且同样的错误不再第二次出现。

## 7. 设计优先级

系统有明确的视觉与实现参照优先级：

1. **设计工具中的设计稿**：最高权威
2. **`Design-Brief.md`**：次级权威
3. **`Product-Spec.md`**：功能逻辑依据

### 7.1 UI 变更的强制检查清单

新增或修改 UI 功能时必须依次完成：

1. 更新 Product Spec
2. 更新设计稿
3. 更新 DEV-PLAN
4. 实现代码
5. 在 review 时对照设计稿验证

如果实现和设计冲突，默认以设计图为准；若设计图确实过时，再更新设计稿并同步记录原因。

## 8. 工作流程定义

### 8.1 标准流程

1. 输入想法
2. 生成 Product Spec
3. 生成 Design Brief
4. 生成设计图
5. 生成 DEV-PLAN
6. 执行项目开发
7. 自动审查
8. 自动修复
9. 提交推送
10. Phase 验证
11. 迭代修改
12. 构建发布
13. 经验沉淀

### 8.2 每一阶段的退出条件

- **Product Spec**：目标、范围、约束、验收标准明确
- **Design Brief**：视觉方向、交互风格、约束清楚
- **设计图**：能指导实现，且与 Spec 一致
- **DEV-PLAN**：任务可执行、依赖明确、可跨 session 续接
- **项目开发**：每个 Task 通过局部编译验证
- **自动审查**：问题清单收敛到可接受范围
- **自动修复**：审查问题已关闭
- **提交推送**：版本状态可追踪
- **Phase 验证**：阶段验收通过
- **构建发布**：产物可交付
- **经验沉淀**：规则与复盘已记录

## 9. 推荐的文档与配置结构

当前采用精简版 `.claude/` 结构，避免模板膨胀：

```text
.claude/
├── agents/
│   ├── code-reviewer.md
│   ├── evolution-runner.md
│   ├── feedback-observer.md
│   └── implementer.md
├── feedback/
│   ├── FEEDBACK-INDEX.md
│   ├── FEEDBACK-LOG.md
│   └── EVOLUTION.md
├── hooks/
│   ├── auto-push.sh
│   ├── check-evolution.sh
│   ├── detect-feedback-signal.sh
│   ├── mark-review-needed.sh
│   ├── pre-commit-check.sh
│   └── stop-gate.sh
├── skills/
│   ├── bug-fixer/
│   ├── code-review/
│   ├── design-brief-builder/
│   ├── design-maker/
│   ├── dev-builder/
│   ├── dev-planner/
│   ├── product-spec-builder/
│   └── release-builder/
├── CLAUDE.md
└── EVOLUTION.md
```

原则：

- `agents` 只保留 4 个职责文件
- `hooks` 只保留 6 个确定性传感器
- `skills` 只保留 8 个核心能力
- `feedback` 只保留索引、日志和进化规则
- 不再额外扩展无必要的模板文件

如果后续要正式开发，建议把每个阶段都做成独立文件，方便追踪与复用。

## 10. 实施路线图

### Phase 0：最小可用闭环

目标：跑通最小流程。

交付物：

- 1 套需求模板
- 1 套设计规范模板
- 1 套开发计划模板
- 1 套 Review/Fix 流程
- 1 套基础日志记录机制

### Phase 1：核心 Harness 成型

目标：让 Skill、Hook、Review、Sub-Agent 各司其职。

交付物：

- 8 个核心 Skill 初版
- 4 个 Sub-Agent 初版
- 关键 Hook 初版
- 自动审查与修复闭环

### Phase 2：进化系统接入

目标：把执行经验变成系统能力。

交付物：

- 反馈记录
- 问题聚类
- 规则升级机制
- 新 Skill 生成流程

### Phase 3：发布与规模化

目标：支持多个项目复用。

交付物：

- 构建发布规范
- 版本化流程
- 多项目配置
- 复盘知识库

## 11. 验收标准

系统达到以下标准时，可认为方案有效：

1. 用户只提供想法，也能生成完整阶段产物
2. 每次开发都有明确的 Review → Fix 闭环
3. UI 变更会自动关联设计稿
4. 失败模式能沉淀为规则或 Skill
5. Sub-Agent 之间不会因为上下文污染而互相误导
6. 能在小项目上稳定跑通从需求到发布的完整链路

---

## 12. 风险与应对

### 风险 1：流程过重

应对：先做最小闭环，只保留最关键阶段。

### 风险 2：Skill 过多导致维护复杂

应对：先做核心 8 个 Skill，后续通过进化系统再扩展。

### 风险 3：Review 流程流于形式

应对：使用固定检查项、自动验证脚本和强制修复闭环。

### 风险 4：上下文污染

应对：严格 Sub-Agent 隔离，只允许显式输入输出。

### 风险 5：设计与实现长期漂移

应对：设计图优先级最高，并要求 UI 改动同步更新设计稿。

---

## 13. 下一步建议

如果要正式启动开发，建议按下面顺序推进：

1. 先确认仓库目录结构和技术栈
2. 定义 Product Spec、Design Brief、DEV-PLAN 模板
3. 搭建 Skill 与 Hook 的最小闭环
4. 实现 code-reviewer 与 bug-fixer 的自动循环
5. 增加执行日志和反馈沉淀
6. 再接入进化系统
7. 最后扩展到构建发布与多项目复用

---

## 14. 结论

这套 Agent Harness 的核心不是让 AI “更会写”，而是让 AI “更可控、更可审查、更能持续进化”。

只要把前馈引导、反馈检查、执行隔离和经验进化这四个机制做扎实，就能把 AI 从一次性回答工具，升级成真正可用于工程交付的开发系统。
